# CR5 Asset Provenance

The URDF and mesh files in this directory are vendored from the official Dobot
GitHub organization, not authored in this repo.

- **Source repo:** https://github.com/Dobot-Arm/TCP-IP-ROS-6AXis
- **Path:** `dobot_description/{urdf/cr5_robot.urdf, meshes/cr5/*.dae}`
- **Commit vendored:** `8f2ef927a517e295cfb45d6fb66ca1e035e734e7` (branch `main`)
- **Date vendored:** 2026-06-30
- **License:** MIT, Copyright (c) 2023 Dobot — see [`LICENSE-cr5-upstream`](LICENSE-cr5-upstream) (verbatim copy of the upstream repo's root `LICENSE` file).

## Modifications made to the vendored files

`urdf/cr5_robot.urdf` had its mesh `<geometry>` URIs rewritten from ROS
package-relative form to plain relative filesystem paths, since this repo has
no ROS package resolver:

```
package://dobot_description/meshes/cr5/<file>.dae  →  ../meshes/<file>.dae
```

No geometry, inertial, or kinematic (link/joint origin) data was altered.

`<limit .../>` `velocity="0"` on all six revolute joints (a SolidWorks-exporter
artifact, not a real limit — see below) was changed to `velocity="3.14"`
(180°/s, Dobot's published CR5 joint speed spec, uniform across all six
joints — see https://www.dobot.us/cr5-6-specifications/). This was **not**
optional cleanup: cuRobo's `MotionGenConfig.load_from_robot_config()` reads
joint velocity limits directly from the URDF with no config-level override
(confirmed via `curobo/cuda_robot_model/cuda_robot_generator.py`'s
`_get_joint_position_velocity_limits()` — there is no `cspace.max_velocity`
key, unlike `max_jerk`/`max_acceleration`), and
`curobo/rollout/cost/bound_cost.py`'s `set_bounds()` hard-fails
(`ValueError: Joint velocity limits is zero`) if it isn't fixed. `effort="0"`
was left as-is (see below — not read by cuRobo at all, only by the Isaac Sim
importer, which already overrides it separately).

## Known quirks inherited from the upstream file

- The URDF was generated by the SolidWorks-to-URDF exporter (see the file's
  own header comment) and originally had every `<joint>` at
  `effort="0" velocity="0"`. This is not a real torque/speed limit — it's an
  artifact of the exporter. `effort="0"` is harmless and left unchanged:
  joint actuation in this project is driven by explicit drive-strength
  parameters set at import time (see `scripts/import_cr5.py`), not by this
  URDF value, and cuRobo's own trajectory optimizer never reads `effort` at
  all. `velocity="0"` **did** need fixing (see above) — cuRobo derives its
  own joint velocity limits straight from the URDF's `<limit>` tag, and a
  zero range there is a hard trajopt-setup failure, not just a
  physically-nonsensical value.
- Mesh format is COLLADA (`.dae`), used directly by Isaac Sim's URDF importer.
